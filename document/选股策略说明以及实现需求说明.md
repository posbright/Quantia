## 1. 背景与目标

当前项目已经具备以下基础能力：

1. 综合选股与策略选股能力，已有较多策略条件与选股结果数据。
2. 回测基础能力，已有组合回测引擎、策略编辑页、回测历史页、回测详情页、回测对比页。
3. 模拟交易基础模块，项目内已存在 `paper_trading` 目录与相关页面/接口基础。
4. 测试基础，已有 `tests/test_portfolio_backtest.py`、`tests/test_backtest_metrics.py`、`tests/test_backtest_dashboard_date_range.py` 等回测相关测试。

本次文档优化的核心目标是：

1. 将“策略选股”中的策略沉淀为结构化策略说明文档。
2. 将策略说明映射为可运行、可回测、可测试的策略模板。
3. 统一策略文档、策略实现、回测验证、模拟盘联动之间的口径。
4. 为后续多策略比较、参数优化、模拟盘/实盘扩展预留稳定架构。

因此，本阶段工作的重点不是再做一个新系统，而是将现有能力组织成一条可复制的策略交付链路。

## 4. 总体实施原则

后续所有策略整理、实现与测试，必须遵循以下原则：

1. 文档先行
	任何策略进入模板库前，必须先形成标准化说明文档。

2. 先可回测，再谈调优
	没有稳定回测口径的策略，不进入优化阶段。

3. 禁止未来函数
	仅允许使用信号日及之前已知数据，禁止用未来价格、未来财务数据、未来事件信息生成当日信号。

4. 禁止过拟合导向的硬编码
	不允许为单一时间段或少量样本特意构造过窄参数范围。

5. 说明、实现、测试三者一致
	策略说明中的条件、策略模板中的逻辑、测试中的断言必须一一对应。

6. 与聚宽风格接口保持兼容
	策略模板优先采用当前项目已支持的聚宽风格接口子集，便于互相验证。

7. 单策略独立运行
	每个策略必须可以独立回测、独立测试、独立展示，不依赖其它策略的运行结果。

## 6. 策略文档标准

后续整理“策略选股说明”时，每个策略必须使用统一结构，至少包含以下字段：

### 6.1 基本信息

1. 策略名称
2. 策略分类
	- 技术形态类
	- 趋势突破类
	- 回调反转类
	- 基本面筛选类
	- 复合因子类
3. 策略来源
	- 项目原有策略
	- 页面策略选股条件整理
	- 经验规则补充
4. 适用标的范围
	- A股主板
	- 创业板
	- ETF
	- 指数成份股等

### 6.2 信号定义

1. 买入信号条件
2. 卖出信号条件
3. 若原策略只有选股没有卖出逻辑，需要补充标准退出逻辑
	可选方式：
	- 固定持有 N 天
	- 止盈止损
	- 趋势反转卖出
	- 再平衡调仓
4. 信号触发时点
	- 收盘后生成，次日开盘执行
	- 当日开盘前生成，当日开盘执行
	- 按周调仓
	- 按月调仓

### 6.3 数据口径

1. 使用的数据表/字段
2. 使用的时间窗口
3. 财务字段的公告口径与可用时点
4. 是否依赖缓存历史数据或衍生指标表

### 6.4 风险约束

1. 是否限制单票仓位
2. 是否限制最大持仓数
3. 是否限制行业集中度
4. 是否过滤 ST、停牌、上市天数不足、成交额过低股票

### 6.5 回测配置建议

1. 推荐回测区间
2. 推荐初始资金
3. 推荐基准指数
4. 推荐调仓频率
5. 推荐手续费、印花税、滑点参数

### 6.6 风险提示

1. 该策略可能失效的市场环境
2. 容易被误用的参数
3. 是否对数据质量特别敏感

## 7. 策略实现标准

将策略文档转换为策略模板时，必须满足以下要求：

### 7.1 模板命名

1. 模板名称与“策略选股”中的策略名称一致。
2. 模板描述中注明来源、版本、适用范围。

### 7.2 模板接口风格

优先使用当前项目已支持的聚宽风格接口子集，例如：

1. `initialize(context)`
2. `handle_data(context, data)`
3. `history()`
4. `order_value()`
5. `order_target()`
6. `order_target_value()`

如果某个策略需要扩展新的上下文能力，应先补能力层，再落策略，不允许在策略代码中硬写数据库读取或绕过沙箱。

### 7.3 模板最小要求

每个模板必须明确：

1. 选股池来源
2. 调仓频率
3. 买入条件
4. 卖出条件
5. 仓位分配方式
6. 风险控制条件

### 7.4 不允许的实现方式

1. 在策略代码中直接访问未来日期数据。
2. 在策略代码中通过文件或网络临时读取未纳入回测口径的数据。
3. 在策略代码中硬编码只对单一股票或单一时间段有效的逻辑，除非该策略本身就是单标的策略。
4. 将策略说明中没有定义的临时规则偷偷加入实现。

## 8. 策略审核规则

策略进入模板库前，必须通过审核。审核维度如下：

### 8.1 可行性审核

1. 策略条件是否能映射到项目已有数据。
2. 是否能够在回测引擎中表达。
3. 是否存在无法确定执行时点的条件。

### 8.2 准确性审核

1. 文档条件与代码条件是否一致。
2. 条件边界是否清楚，例如 `>`、`>=`、窗口长度、基准比较方式。
3. 财务字段是否明确采用可回测口径，而不是事后修正口径。

### 8.3 量化可交易性审核

1. 是否有明确的买卖逻辑。
2. 是否可以形成订单。
3. 是否考虑停牌、涨跌停、流动性等现实约束。

### 8.4 风险审核

1. 是否存在未来函数。
2. 是否明显过拟合。
3. 是否存在样本内调参过多而缺少样本外验证。

审核结论建议统一分为：

1. `通过`：可进入模板库。
2. `需调整`：文档或参数需修改后重审。
3. `不通过`：不适合当前项目落地。

## 10. 分阶段实施计划

本需求建议分 6 个阶段推进，避免一次性铺太大导致文档、代码、测试不同步。

### Phase 1：策略盘点与说明文档整理

目标：先把现有策略定义梳理清楚。

主要任务：

1. 盘点“策略选股”中当前已有的所有策略。
2. 将每个策略整理成统一格式的说明条目。
3. 标记每个策略的数据依赖、买卖逻辑完整性、可交易性问题。
4. 输出首版 `document/策略选股说明.md`。

交付物：

1. 策略清单总表。
2. 每个策略的标准说明。
3. 每个策略的审核状态。

验收标准：

1. 所有现有策略都能在文档中找到。
2. 每个策略都有明确买入逻辑。
3. 对没有原生卖出逻辑的策略，文档中明确补充退出规则。

### Phase 2：策略模板化落地

目标：把通过审核的策略转为可运行模板。

主要任务：

1. 将通过审核的策略逐个实现为模板策略。
2. 在策略列表中沉淀模板，名称与原策略一致。
3. 补齐必要的模板元数据，例如描述、推荐参数、默认回测区间。

交付物：

1. 策略模板库第一批可运行模板。
2. 模板映射表，记录“文档条目 -> 模板名称 -> 实现版本”。

验收标准：

1. 每个模板可在编辑页直接运行回测。
2. 模板不依赖人工补充临时代码才能运行。
3. 模板逻辑与文档描述一致。

### Phase 3：测试与验证闭环

目标：保证策略模板可运行、可复现、可独立验证。

主要任务：

1. 为每个模板增加最小运行测试。
2. 增加未来函数与可疑逻辑检测测试。
3. 增加典型回测区间的基线快照测试或指标阈值测试。
4. 验证新模板不影响已有回测模块。

交付物：

1. 策略模板测试集。
2. 回测验证报告。

验收标准：

1. 每个模板至少有 1 个独立测试用例。
2. 回测测试通过率达到预期。
3. 现有回测与看板相关测试不回归。

### Phase 4：策略对比与展示增强

目标：提升策略间的横向比较能力。

主要任务：

1. 补齐回测结果跨策略对比字段。
2. 规范图表对比与表格对比的口径。
3. 增加收益率、回撤、胜率、盈亏比、交易次数等指标排序。
4. 优化默认排序规则，默认按收益率降序。

交付物：

1. 策略回测比较视图增强。
2. 对比指标口径说明。

验收标准：

1. 用户可以在页面上完成多策略结果对比。
2. 图表与表格数据口径一致。
3. 默认排序逻辑符合预期。

### Phase 5：模拟盘接入与多账户扩展

目标：让已验证策略具备接入模拟盘的统一规范。

主要任务：

1. 定义策略接入模拟盘的最小元数据要求。
2. 支持一个账户运行多个策略，或一个策略运行多个账户实例。
3. 保存交易记录、收益曲线、阶段指标。
4. 提供策略间、账户间的对比能力。

交付物：

1. 模拟盘接入规范。
2. 多策略多账户状态管理方案。
3. 模拟盘展示页增强方案。

验收标准：

1. 模拟盘可查看交易、持仓、收益与主要指标。
2. 支持不同策略之间的对比。
3. 多账户运行时数据不串扰。

### Phase 6：参数调优与历史留痕

目标：支持后续参数优化，但不破坏回测可追溯性。

主要任务：

1. 为每个策略记录参数版本。
2. 记录调优前后回测结果。
3. 支持查看参数历史与结果差异。
4. 为后续更系统的参数寻优预留结构。

交付物：

1. 参数版本模型。
2. 参数调优记录与对比机制。

验收标准：

1. 能够查看某个策略的历史参数版本。
2. 能比较调整前后结果差异。
3. 不覆盖原始基线结果。

## 18. 开发注意事项（本轮实施中发现的关键约束）

以下是在实际开发和测试过程中确认的关键约束，后续所有策略模板和测试编写必须遵守。

### 18.1 avg_cost 不含佣金（与聚宽一致）

`Position._on_buy()` 中 `avg_cost` 只反映成交价格的加权平均，**不包含佣金和滑点费用**。
这与聚宽 JoinQuant 的行为一致。

- 滑点体现在成交价格本身（`actual_price = exec_price * (1 + slippage)`）
- 佣金通过扣减现金体现，不计入 avg_cost
- 所有测试断言必须遵循此规则，不得期望 avg_cost 包含佣金

### 18.2 策略代码中的 `import jqdata`

所有策略模板应在头部包含 `import jqdata`，这是聚宽兼容的标准写法。
本地回测引擎会将其映射为空模块（shim），不会报错。
这样策略代码可以同时在聚宽平台和本地引擎中运行。

### 18.3 `__import__` 是 `safe_import` 存根

沙箱 `_create_safe_namespace()` 中 `__import__` 是受控的 `_safe_import`，不是原生 `__import__`。
测试中不应将 `__import__` 视为"危险内置函数"。

### 18.4 数据库连接重试默认值

`_DB_CONN_RETRIES` 默认值为 1（不重试），适合本地开发环境。
生产环境通过 `QUANTIA_DB_CONN_RETRIES=3` 环境变量覆盖。

### 18.5 模板代码中的股票代码格式

- 聚宽格式带后缀：`000001.XSHE`、`600036.XSHG`
- 本地引擎会自动去除后缀进行匹配
- 模板代码统一使用聚宽格式，确保跨平台兼容

### 18.6 attribute_history 与 history 的区别

- `history(code, count, field)` — 返回单字段 Series
- `attribute_history(code, count, unit, fields)` — 返回多字段 DataFrame
- 需要多字段数据时（如 close + volume），必须用 `attribute_history`

### 18.7 init_job 基础表数量

`create_new_base_table()` 当前创建 3 张表（`cn_stock_attention`、`cn_stock_trade_date`、`cn_stock_financial`），相关测试断言需保持同步。

### 18.8 模拟交易数据源原则

模拟交易中除账户金额是虚拟的外，所有数据均为实盘数据：

- **K线行情**：pickle 缓存 → `cn_stock_spot` 数据库 → EastMoney API（三级降级）
- **基本面数据**：`cn_stock_selection` 表（70+ 列真实 PE/PB/ROE/毛利率/负债率/增长率等）
- **禁止合成数据**：模拟交易的 `get_fundamentals()` 不使用哈希合成值，直接读取 `cn_stock_selection`

回测引擎的 `FundamentalDataProvider` 仍保留合成数据兜底（用于历史日期无真实数据的场景），但模拟交易引擎优先使用真实数据。

### 18.9 cn_stock_selection 字段映射

模拟交易 `get_fundamentals()` 中聚宽字段名与 `cn_stock_selection` 列名的映射关系：

| 聚宽字段 | cn_stock_selection 列 | 说明 |
|---------|----------------------|------|
| market_cap | total_market_cap（÷1e8） | 总市值（元→亿元） |
| pe_ratio | pe9 | 市盈率 TTM |
| pb_ratio | pbnewmrq | 市净率 MRQ |
| circulating_market_cap | free_cap（÷1e8） | 流通市值（元→亿元） |
| roe | roe_weight | 净资产收益率 |
| eps | basic_eps | 每股收益 |
| gross_profit_margin | sale_gpr | 毛利率 |
| net_profit_margin | sale_npr | 净利率 |
| inc_total_revenue_year_on_year | toi_yoy_ratio | 营收同比增长率 |
| inc_net_profit_year_on_year | netprofit_yoy_ratio | 净利润同比增长率 |
| total_liability/total_assets | debt_asset_ratio | 资产负债率（百分比） |
| net_operate_cash_flow | per_netcash_operate | 每股经营现金流 |

### 18.10 模拟交易 initialize 执行时机

模拟交易中 `initialize(context)` **每次运行都执行**（非仅首次），因为 `run_daily`/`run_weekly` 回调列表需要每次重新注册。这与聚宽回测中 `initialize` 只执行一次不同，是模拟交易的特殊需求。
