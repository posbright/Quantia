## 二、K线缓存更新（核心请求源）

### 2.1 调用链路

```
execute_daily_job → kline_cache_daily_job.fetch_all_data()
  → stockfetch.update_all_caches(stocks, date_start, date_end)
    → _update_one(stock)  [线程池并发]
      → _read_cache_meta(code)  [元数据预检查]
      → stock_hist_cache_incremental(code, ...)
        → _fetch_from_sources(code, fetch_start, date_end)
          → stock_zh_a_hist()        [东方财富]  -- 主数据源
          → stock_zh_a_hist_tencent() [腾讯]     -- 降级备选
          → stock_zh_a_hist_sina()    [新浪]     -- 最终备选
```

### 2.2 本地模式 vs 服务器模式

| 参数 | 本地模式 | 服务器模式 | 环境变量覆盖 |
|------|---------|-----------|------------|
| 默认线程数 | 6 | 2 | `QUANTIA_KLINE_CACHE_WORKERS` |
| 最大线程数 | 12 | 4 | — |
| 请求间隔（秒） | 0.2-0.5 | 1.0-3.0 | `QUANTIA_KLINE_REQUEST_DELAY_MIN/MAX` |
| 批次大小 | 300 | 100 | `QUANTIA_KLINE_CHUNK_SIZE` |
| 批次暂停（秒） | 2-4 | 8-15 | `QUANTIA_KLINE_BATCH_PAUSE_MIN/MAX` |
| 连续失败阈值 | 5 | 3 | — |
| 限流暂停基准（秒） | 60 | 120 | — |
| 最大限流次数 | 5（然后熔断） | 3（然后熔断） | — |
| 启用标志 | `QUANTIA_LOCAL_MODE=1` | 默认 | `QUANTIA_LOCAL_MODE` |

### 2.3 五层限流防护

```
┌─────────────────────────────────────────────────────────────┐
│ 第1层 — 并发控制                                              │
│   线程数: 本地6(最大12) / 服务器2(最大4)                        │
│   作用: 控制同一时刻并行请求数                                   │
├─────────────────────────────────────────────────────────────┤
│ 第2层 — 请求间隔                                              │
│   每只股票处理完成后: 本地0.2-0.5s / 服务器1.0-3.0s              │
│   仅对实际发起API请求的股票生效（skip不延迟）                     │
├─────────────────────────────────────────────────────────────┤
│ 第3层 — 批次冷却                                              │
│   每处理N只股票暂停: 本地每300只暂停2-4s / 服务器每100只暂停8-15s  │
│   仅在有API请求时暂停（全部skip的批次跳过）                       │
│   暂停时执行 gc.collect() 释放内存                               │
├─────────────────────────────────────────────────────────────┤
│ 第4层 — 限流检测 & 暂停                                        │
│   连续失败N次触发: 本地5次 / 服务器3次                            │
│   指数退避暂停: 本地 60→120→240→480→960s / 服务器 120→240→480s   │
│   恢复后自动降速: 请求间隔 ×1.5（上限5-8秒）                      │
├─────────────────────────────────────────────────────────────┤
│ 第5层 — 熔断保护                                              │
│   限流触发次数达上限: 本地5次 / 服务器3次 → 终止整个任务             │
│   避免持续请求导致 IP 被永久封禁                                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.4 请求数估算

**场景A：日常增量更新（缓存基本最新）**

| 步骤 | 说明 | 请求数 |
|------|------|--------|
| meta 预检查 | ~5000只 × 读取 .meta 文件 | 0（磁盘I/O） |
| skip（缓存已最新） | 约80-95%的股票跳过 | 0 |
| 增量获取 | 约250-1000只需更新 | 250-1000 |
| **合计** | | **250-1000** |

⏱ 耗时估算（本地模式）：500只 × (0.35s延迟 + 0.5s请求) / 6线程 ≈ **70秒**
⏱ 耗时估算（服务器模式）：500只 × (2s延迟 + 0.5s请求) / 2线程 ≈ **625秒（~10分钟）**

**场景B：首次全量获取（无缓存）**

| 步骤 | 说明 | 请求数 |
|------|------|--------|
| 无缓存 | 全部5000只需请求 | 5000 |
| 数据源重试 | 约5%失败切换源 | +250 |
| **合计** | | **~5250** |

⏱ 耗时估算（本地模式）：5250 × 0.85s / 6线程 ≈ **12分钟**（不含限流暂停）
⏱ 耗时估算（服务器模式）：5250 × 2.5s / 2线程 ≈ **109分钟（~1.8小时）**

### 2.5 ⚠️ 双重延迟问题

目前存在**双重延迟叠加**：

1. **爬虫模块内部延迟**（`stock_hist_em.py`）：每次请求前 `time.sleep(0.2-0.5)`
2. **调度层延迟**（`_update_one`）：每只股票处理后 `time.sleep(0.2-0.5)`

实际每只股票延迟 = 0.2-0.5(爬虫) + 0.2-0.5(调度) = **0.4-1.0秒**（本地模式）

| 数据源 | 爬虫内部延迟 | + 调度层延迟 | = 实际总延迟 |
|--------|------------|------------|------------|
| 东方财富 | 0.2-0.5s | +0.2-0.5s | **0.4-1.0s** |
| 腾讯财经 | 0.3-0.8s (短) / 0.5-1.5s (长) | +0.2-0.5s | **0.5-2.0s** |
| 新浪财经 | **3-6s** | +0.2-0.5s | **3.2-6.5s** |

### 配置参数
| 参数 | 默认值 | 环境变量 |
|------|--------|----------|
| 连续失败阈值 | 5次 | `DATA_SOURCE_FAIL_THRESHOLD` |
| 冷却时间 | 300秒（5分钟） | `DATA_SOURCE_COOLDOWN_SECONDS` |
| 最大冷却时间 | 3600秒（1小时） | `DATA_SOURCE_MAX_COOLDOWN` |

### 降级机制
```
正常 → 连续5次失败 → 降级（排到最后）→ 冷却300秒
      → 再失败 → 冷却600秒 → 再失败 → 冷却1200秒 → ... → 最大3600秒
      → 冷却结束 → 恢复正常优先级
```

### 数据源选择顺序
```python
_fetch_from_sources 尝试顺序:
1. 东方财富（主数据源，数据最全）
2. 腾讯财经（备选，数据格式不同）
3. 新浪财经（最终备选，限流最严格）

# 降级后的源排到末尾，但仍会被尝试
# 连接级错误（IP封禁/网络断开）立即跳到下一源，不重试
```

## 五、代理池机制

### 请求路由策略
```
东方财富请求:
  ├── 代理池 ≥10: 30%直连 / 70%代理
  ├── 代理池 3-9: 60%直连 / 40%代理
  └── 代理池 <3:  80%直连 / 20%代理

每次请求3次重试:
  第1-2次: 使用代理（失败换不同代理）
  第3次:   强制直连（代理全部失败时的保底）
```

### 代理池维护
| 操作 | 频率 | 并发 |
|------|------|------|
| 初始化验证 | 启动时一次 | 50线程 |
| 定期刷新 | 每10分钟 | 50线程 |
| 紧急补充 | 池<3时触发 | 50线程，30秒冷却 |

## 六、优化建议

### 🔴 高优先级

#### 1. 消除双重延迟（预计提速 30-40%）

**问题**：东方财富历史K线存在双重延迟叠加

- `stock_hist_em.py:283` 在请求前添加 0.2-0.5s 延迟
- `_update_one` 在请求后又添加 0.2-0.5s（本地）/ 1.0-3.0s（服务器）延迟

**方案**：在 `_update_one` 层统一控制延迟，移除爬虫模块内的 `time.sleep`。
好处：延迟策略集中管理，避免重复；且调度层已有完善的5层限流机制。

**预期效果**：本地模式每只股票实际延迟从 0.4-1.0s 降至 0.2-0.5s，等效提速约 50%。

#### 2. 增量请求的精确范围（预计减少 20-30% 数据传输）

**问题**：`stock_hist_cache_incremental` 已计算出精确的 `fetch_ranges`（仅获取缺失日期段），但部分数据源API始终返回全量数据（不尊重 start_date）。

**方案**：对于增量更新场景（缓存仅缺最近1-3天），优先使用支持精确日期范围的API。东方财富 API 的 `beg`/`end` 参数已支持此功能，确认其生效。

### 🟡 中优先级

#### 3. 增加本地模式跳过率

**方案**：在 `_update_one` 中增加基于文件修改时间的快速检查。如果 `.pickle` 文件的 mtime 在今日范围内，跳过 meta 文件的读取解析。

#### 4. 新浪数据源延迟优化

**问题**：新浪历史K线延迟 3-6s/只，是东方财富的 6-12 倍。
**现状**：仅在前两个源都失败时才使用，风险可控。
**方案**：考虑动态降低新浪延迟至 1-2s，但增加更灵敏的限流检测（监测 456 状态码）。

#### 5. 数据源重试间隔优化

**问题**：`_retry_sleep` 基准 90 秒太长（第1次重试等 99-117 秒，第2次 198-234 秒）。
**方案**：本地模式使用更短的重试间隔（如 30 秒基准），服务器模式保持 90 秒。

### 🟢 低优先级

#### 6. 批量历史K线API

**问题**：东方财富历史K线不支持批量获取（逐只请求）。
**方案**：调研是否存在批量API（如通过 WebSocket 或特殊接口一次获取多只K线），目前未发现公开的批量K线接口。

#### 7. 缓存预热策略

**方案**：工作日收盘后（15:30-16:00）自动触发增量更新，利用API负载最低时段获取数据。可在 cron 配置中设置。
