## 1. 风险矩阵

### 1.1 风险评级标准

| 等级 | 影响 | 概率 | 优先级 |
|---|---|---|---|
| 🔴 **严重** | 项目无法按期交付 或 生产故障 | 中 | P0 |
| 🟠 **高** | 关键功能异常 或 交付延期 | 中高 | P1 |
| 🟡 **中** | 部分功能缺陷 或 小延期 | 中 | P2 |
| 🟢 **低** | 影响轻微 或 易规避 | 低 | P3 |

## 2. 风险详表

### 2.1 技术风险

#### R1: 跨语言网络分裂导致数据不一致 (🔴 严重)

| 属性 | 值 |
|---|---|
| **场景** | Python web 发送通知到 MQ,但 Java 消费端网络中断 3 小时 |
| **影响** | 通知堆积,用户收不到模拟交易成交通知 |
| **概率** | 低(IDC 网络稳定) 但非零 |
| **缓解** | ① 本地数据库备份队列(notification_event 表 status='pending') ② 自动重试(RMQ TTL 48h) ③ 告警立即通知运维 ④ 降级策略:通知失败后改用邮件通知 |
| **监控** | 每分钟检查 RMQ 队列深度、Python-Java 网络延迟 |

#### R2: RabbitMQ 宕机 (🔴 严重)

| 属性 | 值 |
|---|---|
| **缓解** | ① RabbitMQ 集群部署(3 节点),自动 failover ② MySQL 本地缓存作备份 ③ 部署 RabbitMQ 监控告警(queue depth > 10000) ④ 故障恢复 SOP(2 小时内修复) |
| **成本** | 额外部署 2 个 RabbitMQ 节点(+2 台虚机) |

#### R3: Kafka 分区不均衡导致交易延迟 (🟠 高)

| 属性 | 值 |
|---|---|
| **场景** | Kafka 某个分区 lag 堆积(消费速度跟不上生产) |
| **影响** | 某些股票的交易命令延迟 5 分钟以上 |
| **缓解** | ① Kafka 分区按 code % 3 分片,保证同股票顺序性 ② 消费端多线程(10 threads) ③ 自动扩容(lag > 1000 自动告警) ④ 参数调优(batch.size, linger.ms) |

#### R4: Python-Java 数据一致性无法保证 (🟠 高)

| 属性 | 值 |
|---|---|
| **场景** | Python job INSERT 信号后 crash,Java 消费方读到半成品数据 |
| **缓解** | ① 所有操作幂等(unique constraint on command_id) ② 消息版本化(schema_version) ③ T+1 日志对账脚本 ④ 双向校验(Python → Java 通知后,Java 返回 ACK) |

#### R5: LLM API 超时堵塞交易 (🟠 高, 仅 ai-decision-svc)

| 属性 | 值 |
|---|---|
| **场景** | OpenAI API 响应时间 > 30s,live-trade-svc 等待超时 |
| **缓解** | ① 异步调用 AI(不 block 交易) ② 降级策略:AI 超时时自动 PASS ③ 缓存 prompt (Redis TTL=1h) ④ 熔断器(Resilience4j,失败 3 次后打开) |

#### R6: 共享库导致跨服务 JOIN 失败 (🟠 高)

| 属性 | 值 |
|---|---|
| **场景** | Python 写入 cn_stock_strategy_* 后,Java 需要 JOIN cn_stock_notification_event |
| **缓解** | ① 禁止跨边界 JOIN(编码规范) ② Code Review 严格检查 ③ 短期(6 个月)允许共享库,但明确表所有权 ④ 长期(1 年后)逐步拆库 |

### 2.2 业务风险

#### R7: 通知丢失导致用户不知道成交结果 (🔴 严重)

| 属性 | 值 |
|---|---|
| **当前问题** | Python 轮询方案脆弱:Python 服务重启 → 轮询中断 |
| **缓解** | ① MQ 引入后大幅改善(at-least-once 语义) ② 重试 3 次 + DLQ ③ 告警通知运维 ④ 手动发送补偿通知 |

#### R8: 交易误执行 (生产级别风险) (🔴 严重)

| 属性 | 值 |
|---|---|
| **场景** | IM 指令 "buy 100 shares" 被重复执行 2 次 |
| **缓解** | ① **幂等性**:command_id unique constraint ② 数据库级唯一约束 ③ IM 指令去重(dedupe_key) ④ 严格测试(模拟 Kafka 分区重平衡) ⑤ T+1 对账检查 Broker 是否有重复成交 |

#### R9: 回滚困难 (🟠 高)

| 属性 | 值 |
|---|---|
| **问题** | 一旦上线 Java 微服务,若出现严重 bug,回滚路径不清 |
| **缓解** | ① 灰度上线(5% → 25% → 50% → 100%) ② 保留 Python 原始实现作 fallback ③ 特性开关(Feature Flag)快速切换 ④ 自动化回滚脚本 |

### 2.3 人员风险

#### R10: 团队缺乏 Spring Cloud 经验 (🟠 高)

| 属性 | 值 |
|---|---|
| **缓解** | ① 外聘 Spring Cloud 顾问(2-4 周 ramp-up) ② 内部培训 + Code review ③ 使用成熟框架(不自己造轮子) ④ 文档完善(API 设计、运维 SOP) |

#### R11: 知识流失 (🟡 中)

| 属性 | 值 |
|---|---|
| **问题** | Java 开发者离职,新人不了解微服务架构 |
| **缓解** | ① 详细架构文档 ② 代码注释规范 ③ Runbook 维护 ④ 定期知识分享 |

### 2.4 基础设施风险

#### R12: 部署成本增加 (🟡 中)

| 属性 | 值 |
|---|---|
| **额外基础设施** | RabbitMQ x3, Kafka x3, Zipkin x1, Prometheus x1, Grafana x1 |
| **成本估算** | 约 10 台虚机(或容器编排) |
| **缓解** | ① 云托管(AWS MSK, Alibaba MQ) ② 容器化部署(Kubernetes) ③ 资源优化(pod 资源限制) |

#### R13: 监控成本 (🟡 中)

| 属性 | 值 |
|---|---|
| **工作量** | Prometheus + Grafana + ELK 配置(1-2 周) |
| **缓解** | ① 复用开源方案 ② 使用商业监控(DataDog, New Relic) ③ 自动化告警规则 |

## 3. 风险缓解计划

### 3.1 优先级排序

```
P0 (立即处理):
  R1: 跨语言网络分裂 → 实现本地缓存队列
  R2: RabbitMQ 宕机   → 部署 3 节点集群
  R7: 通知丢失       → MQ + 重试机制
  R8: 交易误执行     → 幂等性 + 对账

P1 (Phase 1/2 完成前):
  R3: Kafka 分区不均 → 监控 + 自动扩容
  R4: 数据一致性     → 版本化 + 对账脚本
  R10: 团队经验      → 外聘顾问 + 培训

P2 (后续优化):
  R5: LLM API 超时   → 异步 + 降级
  R9: 回滚困难       → 灰度 + Feature Flag
  R11: 知识流失      → 文档 + 分享
  R12/R13: 成本      → 云托管 + 自动化
```

## 4. 故障模式分析 (FMEA)

| 故障模式 | 影响 | 检测 | 缓解 | 优先级 |
|---|---|---|---|---|
| RMQ queue 满 | 通知全失 | 告警(depth>10K) | 自动缩放 | P0 |
| DB 连接耗尽 | 服务挂 | 监控(active conn) | 连接池调优 | P0 |
| MQ 消息重复 | 重复交易 | 自动检测(dedupe_key) | 幂等性 | P0 |
| Python 网络分裂 | 单向通信失败 | 心跳检测 | 本地缓存 | P1 |
| Kafka lag 高 | 交易延迟 | Consumer lag 监控 | 扩容消费者 | P1 |
| LLM API 慢 | 交易变慢 | 响应时间 SLA | 异步 + 缓存 | P2 |

## 5. 应急预案

### 5.1 通知中断应急流程

```
触发条件: notification-svc 宕机 || RabbitMQ 宕机 5 分钟

1. 自动告警(PagerDuty) → 值班运维
2. 检查:
   - RabbitMQ 集群状态 (rabbitmqctl status)
   - Java 应用日志 (kubectl logs notification-svc)
   - 数据库连接
3. 处理:
   - 如果 RabbitMQ 故障: 手动 failover 到备用节点
   - 如果 Java app 宕机: 自动重启 + 验证健康检查
   - 清理堆积消息 (手动处理 DLQ)
4. 恢复验证:
   - 发送测试消息 → 验证通知收到
   - 检查 notification_event 表 status='sent' 比例 > 95%
5. 事后:
   - Post-mortem 分析根本原因
   - 更新监控告警规则
```

### 5.2 交易中断应急流程

```
触发条件: live-trade-svc 宕机 || 下单失败率 > 10%

1. 自动告警 + 立即停止 IM 指令接收(QUANTIA_IM_COMMAND_ENABLED=0)
2. 检查:
   - live-trade-svc 日志
   - Broker API 连通性
   - Kafka 消费 lag
3. 处理:
   - 手动下单(通过 Broker 官方客户端)
   - 对账现有成交单
   - 补偿失败的交易(告警通知用户)
4. 恢复:
   - 启用 IM 指令接收(QUANTIA_IM_COMMAND_ENABLED=1)
   - 验证新指令能正常执行
5. 事后:
   - 分析失败原因
   - 更新对账脚本
```

## 6. 监控与告警规则

### 6.1 关键指标

| 指标 | 告警阈值 | 告警级别 | 响应时间 |
|---|---|---|---|
| **Notification TPS** | < 500 | 🟠 Orange | 5 min |
| **RabbitMQ queue depth** | > 10000 | 🔴 Critical | 2 min |
| **Kafka consumer lag** | > 1000 | 🟠 Orange | 10 min |
| **Database active connection** | > 80% | 🟠 Orange | 5 min |
| **Trade execution success rate** | < 95% | 🔴 Critical | 2 min |
| **API response time P99** | > 5s | 🟡 Yellow | 10 min |
| **Java heap memory** | > 85% | 🟡 Yellow | 10 min |

### 6.2 告警通道

- **Critical (🔴)**: PagerDuty → SMS + 电话
- **Orange (🟠)**: Slack + Webhook
- **Yellow (🟡)**: Email + Slack
