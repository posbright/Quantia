## 🎯 关键要点提炼

### 核心结论 (1 句话)
> **外围异步链路 Java 化(4 个微服务) + 核心计算保留 Python,通过 MQ 解耦,改造周期 6 个月,风险可控。**

### 改造范围
✅ **改造(Java)**:
- notification-svc (通知推送)
- im-gateway-svc (IM 指令网关)
- ai-decision-svc (可选, LLM 决策)
- live-trade-svc (高风险,需谨慎)

❌ **保留(Python)**:
- core/ (计算引擎)
- web/ (API 网关)
- job/ (定时任务)
- paper_trading, trade (模拟盘、事件驱动)

### 关键数据
| 指标 | 值 |
|---|---|
| 总周期 | 24 周 (6 个月) |
| 工作量 | 500 人天 |
| 团队规模 | 11 人 |
| 基础设施成本 | ~10 台虚机 |
| 综合可行性评分 | 7.8/10 (中等偏高) |
| 建议 | ✅ 批准启动 |

### 关键风险 (已缓解)
1. 📞 跨语言网络分裂 → 本地缓存 + 自动重试
2. 🔥 RabbitMQ 宕机 → 3 节点集群 + failover
3. 💔 通知丢失 → MQ 可靠投递 + DLQ
4. 🎰 交易误执行 → 幂等性 + 对账 + Saga
5. ⏸️ LLM 超时 → 异步 + 降级

### 单机小内存服务器可行性结论

- 结论: **可行,但仅适合 PoC/小流量生产(通知+IM,不含 live-trade 全量)。**
- 推荐运行范围: 仅 `notification-svc` + `im-gateway-svc` + MySQL + RabbitMQ + 基础监控。
- 不建议在单机同时跑 Kafka + 全套监控 + 高并发交易执行。

最低配置(可跑通):
- CPU: 4 vCPU
- 内存: 8 GB
- 磁盘: 120 GB SSD

建议配置(稳态运行):
- CPU: 8 vCPU
- 内存: 16 GB
- 磁盘: 200 GB SSD

说明:
- 8 GB 内存需要关闭部分组件(如 ELK、全量 Grafana 大盘)并限制 JVM 堆。
- 若上 `live-trade-svc` 或高峰消息量，建议升级到多机或托管 MQ。

## 📞 快速问答

**Q: 为什么不全部改成 Java?**  
A: core/ 依赖 pandas/numpy/talib/backtrader/akshare,Java 无等效物,迁移=重写,风险太高。选择外围异步链路改造,性价比最优。

**Q: Spring Cloud 必须上吗?**  
A: 不必须全部上。精选 Gateway+Config+Zipkin+MQ 即可(跨语言友好)。Seata/Eureka 因跨 Python 支持差,暂不上。

**Q: 如果出现严重 bug 怎么办?**  
A: 灰度上线(5%→25%→50%→100%)+ Feature Flag 快速切换。任何时刻都能回滚到 Python 原始实现。

**Q: 需要多少人?**  
A: 核心 4-5 人 + 支持人员 6-7 人。其中 QA 占比 30%,DevOps 常驻监控。

**Q: 成本多少?**  
A: 开发 ~100 万人民币 + 基础设施 ~20-30 万/年。ROI: 运维效率↑40%, 稳定性↑30%, 1 年回本。
