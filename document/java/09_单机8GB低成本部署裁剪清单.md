# Quantia Java 微服务改造 — 单机 8GB 低成本部署裁剪清单

**文档版本**: 1.0  
**日期**: 2026-06-11  
**目标**: 在单台 8GB 服务器上以最低成本落地，并为后期扩容与持续迭代预留演进路径。

---

## 1. 结论先行

- **可行性结论**: 可行，但建议用于 PoC、内测或低流量生产。
- **推荐上线范围**: `notification-svc` + `im-gateway-svc` + `MySQL` + `RabbitMQ` + 轻量监控。
- **不建议同机长期承载**: Kafka 高吞吐、ELK 全套、live-trade 高峰交易。
- **最低硬件**: 4 vCPU / 8 GB RAM / 120 GB SSD。
- **建议硬件**: 8 vCPU / 16 GB RAM / 200 GB SSD。

---

## 2. 单机 8GB 组件裁剪清单

### 2.1 必选组件 (上线最小闭环)

- [ ] MySQL 8.0 (单实例)
- [ ] RabbitMQ 3.11 (单节点, 开启持久化)
- [ ] notification-svc (Java)
- [ ] im-gateway-svc (Java)
- [ ] Python web/core/job (保留)
- [ ] 基础监控: Prometheus(精简抓取) + Node Exporter

### 2.2 延后组件 (资源吃紧时先不上)

- [ ] Kafka (仅开发/演示使用, 生产单机不建议)
- [ ] Grafana 大盘(可先用 Prometheus + 简单告警)
- [ ] Zipkin 全量追踪(先采样 5%-10%)
- [ ] ELK 全套(日志先落盘 + logrotate)
- [ ] ai-decision-svc / live-trade-svc

### 2.3 明确禁配

- [ ] 单机同时运行 Kafka + ELK + 全量 Zipkin + 交易高峰
- [ ] MySQL 与 RabbitMQ 共用同一高 IO 小盘(<100GB)
- [ ] 不设日志轮转与磁盘告警

---

## 3. 单机资源预算 (8GB 内存)

## 3.1 内存分配建议

| 组件 | 建议内存上限 | 说明 |
|---|---:|---|
| MySQL | 2.0 GB | 限制 buffer pool，避免挤压应用 |
| RabbitMQ | 1.0 GB | 开启高水位控制 |
| notification-svc | 1.0 GB | JVM `-Xms256m -Xmx768m` |
| im-gateway-svc | 1.0 GB | JVM `-Xms256m -Xmx768m` |
| Python web/core/job | 1.0 GB | 低峰策略任务优先 |
| Prometheus + node-exporter | 0.8 GB | 仅保留关键指标 |
| OS 预留 | 1.2 GB | 避免 OOM 与抖动 |
| **合计** | **8.0 GB** | 需要严格限额 |

### 3.2 CPU 与磁盘建议

| 项目 | 最低 | 建议 |
|---|---:|---:|
| CPU | 4 vCPU | 8 vCPU |
| SSD | 120 GB | 200 GB |
| IOPS | >2000 | >5000 |

## 4. 单机部署参数基线 (可直接套用)

### 4.1 JVM 基线

- notification-svc: `-Xms256m -Xmx768m -XX:+UseG1GC -XX:MaxGCPauseMillis=200`
- im-gateway-svc: `-Xms256m -Xmx768m -XX:+UseG1GC -XX:MaxGCPauseMillis=200`

### 4.2 MySQL 基线

- `innodb_buffer_pool_size=1G`
- `max_connections=200`
- `slow_query_log=ON`
- 开启每日备份 + binlog

### 4.3 RabbitMQ 基线

- durable queue + message persistence
- `vm_memory_high_watermark.relative=0.5`
- DLQ 必开，TTL 必配

### 4.4 日志基线

- 应用日志保留 7 天，超过自动压缩/清理
- 单文件上限 200MB，滚动 10 份
- 磁盘使用率 >80% 报警

## 5. 上线前检查清单 (单机版)

- [ ] 关键链路压测通过: notification P99 < 5s
- [ ] RabbitMQ 消息可重试且可入 DLQ
- [ ] MySQL 慢查询日志已开启
- [ ] JVM 堆限制已生效
- [ ] 日志轮转与磁盘告警已配置
- [ ] 回滚开关已验证(可切回 Python 路径)
- [ ] 备份恢复演练完成(至少一次)

---

## 6. 扩容触发器 (什么时候必须升级)

满足任一条，建议 2 周内启动扩容:

- [ ] CPU 平均 > 70% 且持续 15 分钟(每天 >= 3 次)
- [ ] 内存使用 > 85% 且出现 swap
- [ ] RabbitMQ 队列深度 > 5000 持续 10 分钟
- [ ] MySQL 慢查询 > 200/小时
- [ ] 通知成功率 < 95% 或 P99 > 8s
- [ ] 单机月度故障 > 2 次

---

## 7. 后期扩容路径 (低风险演进)

### 阶段 A: 单机稳态 (当前)

- 架构: 单机全部核心组件
- 目标: 先稳定业务闭环，低成本上线

### 阶段 B: 先拆状态层 (优先)

- 将 MySQL 迁移到独立主机/托管 RDS
- RabbitMQ 迁移到托管或 2 节点 + 1 仲裁
- 应用仍可单机

### 阶段 C: 拆计算层

- notification-svc / im-gateway-svc 分离到独立节点
- 接入统一 API Gateway
- 增加灰度发布能力

### 阶段 D: 高可用化

- Kafka 集群化(3 broker)
- K8s 或至少双机热备
- 可观测性升级(Prometheus + Grafana + Trace 全链路)

---

## 8. 持续迭代机制 (避免一次性大改)

### 8.1 双周迭代节奏

- 第 1 周: 稳定性优化(告警、慢查询、重试)
- 第 2 周: 能力迭代(新接口、新策略、性能改进)
- 每次迭代末: 发布复盘 + 指标回顾

### 8.2 每月技术债清理

- [ ] 清理 TOP10 慢查询
- [ ] 清理 TOP10 错误日志
- [ ] 升级高危依赖(CVE)
- [ ] 校验备份可恢复性

### 8.3 变更治理

- 所有新特性必须有 Feature Flag
- 所有跨服务变更必须有回滚脚本
- 所有接口变更必须版本化(`_schema_version`)

---

## 9. 成本与收益评估 (单机方案)

### 9.1 成本

- 硬件/云主机: 1 台 4C8G
- 运维复杂度: 低
- 人员要求: 1 名 DevOps 可维护

### 9.2 收益

- 快速上线验证架构正确性
- 低预算前提下完成 Java 化试点
- 为后续多机扩容保留兼容路径

### 9.3 风险

- 单点故障不可避免
- 资源上限明显
- 高峰期易抖动
