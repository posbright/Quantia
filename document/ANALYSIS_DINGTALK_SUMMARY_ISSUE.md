# 钉钉报告摘要截取分析报告

## 一、为什么只截取前 500 字符？

### 直接原因
**代码位置**：[quantia/job/stock_report_scheduled.py#L383](quantia/job/stock_report_scheduled.py#L383)

```python
summary = content[:500]
push_report_summary_to_dingtalk(code, stock_name, summary, rating)
```

### 500 字符限制的背景

| 维度 | 说明 |
|------|------|
| **钉钉 markdown 硬限制** | `text` 字段最多 2000 字符（API 规范） |
| **移动端展示限制** | 钉钉 APP 移动端对长内容的展示和交互体验很差；通常建议控制在 500-800 字符内 |
| **当前设计选择** | 500 字符是"保守"的移动端优化值，确保在窄屏手机上一屏内不产生复杂排版 |
| **无代码注释** | 代码中没有明确标注为什么是这个数字，属于"隐性知识" |
| **无动态配置** | 硬写在代码里，无法通过偏好表调整 |

### 问题
- **不是功能限制**，而是**体验权衡**：钉钉技术上可以支持 2000 字，但为了移动端效果而被压缩。
- **截断导致的内容损失**：AI 报告开头通常是"标题 + 数据说明 + 技术指标表"，真正的结论和建议在后半段。
- **结构化字段失效加剧**：就算结构化解析成功提取了 `rating`、`short_term_advice` 等，也因为 500 字符限制而被 truncate，消息里仍然显示"—"或"[截断]"。

---

## 二、两种改进方案对比

### 方案 A：直接增加字符限制（**不推荐**）

**改法**：把 `[:500]` 改成 `[:1500]` 或 `[:2000]`

**技术难点**：
- ⚠️ **几乎零难度** 实现，改一行代码
- ❌ **移动端体验恶化**：钉钉 APP 被迫显示超长消息，导致排版混乱、表格错位、滚动困难
- ❌ **没有解决根本问题**：仍然是"报告前 N 字"，结论不一定会出现

**优缺点**：

| 优 | 缺 |
|----|---|
| 实现快 | 移动端体验很差 |
| 代码改动小 | 如果仍是表格/标题开头，信息还是碎 |
| 可逐步调试 | 用户体感变差，可能反而抱怨"太长" |

**结论**：**不推荐** — 虽然快，但彻底解决不了"摘要不聚焦、结论偏后"的问题。

---

### 方案 B：提取结论摘要（**推荐**）

**改法**：不再发 `report_md[:500]`，改成发结构化的 `评级 + 短中长期建议 + 风险提示`，配合 AI 文本提取

**核心逻辑**：
1. 优先发送 DB 中已有的结构化字段（`rating`, `rating_score`, `short_term_advice` 等）
2. 若结构化字段失效（为 NULL），用文本搜索从整篇报告中提取结论段落
3. 组装成 1 段摘要（200-400 字）

**技术难点分解**：

#### 1. 结构化字段的完整性问题 ⭐⭐⭐
**当前现象**：最近几条 cron 报告的 `rating`, `rating_score` 都是 `NULL`

**原因追踪**：
- `report_parser.extract_structured_fields()` 依赖报告里出现特定的结构标记（如"评级""中性""看多"等）
- AI 生成的报告格式不稳定（有时用"综合评级"，有时用"总体判断"，有时没有结论节点）
- 正则表达式匹配失败 → 返回 `None`

**改进难度**：⭐⭐ **中等**
- 需要升级 `report_parser.extract_structured_fields()` 的正则库（加更多同义词）
- 需要给 AI agent 的 system prompt 加强制要求（"必须在最后一节标注 [综合评级] 字段"）
- 需要增加 fallback 路径：若一级提取失败，用 LLM 二次提取（消耗 token，但可靠）

#### 2. Fallback 文本提取逻辑 ⭐⭐⭐⭐
**需要做**：若 `rating` 为 NULL，从报告里自动搜索结论段落

**可选方案**：
- **方案 B1**（简单）：正则搜索"总结""结论""评级"附近的 1-2 段，取前 200 字
  - 难度：⭐⭐ （30 行代码）
  - 可靠性：70%（格式不稳定时会miss）
  
- **方案 B2**（推荐）：调用 LLM 的 extraction API（如 claude 的 tool use）
  - 难度：⭐⭐⭐⭐ （需要额外 API 调用）
  - 可靠性：95%（LLM 理解报告上下文）
  - 成本：~0.01 CNY/调用（可接受）
  - 时延：+0.5–1s（可接受）

- **方案 B3**（混合）：先用正则搜索，失败再用 LLM
  - 难度：⭐⭐⭐
  - 可靠性：98%（兜底完善）
  - 成本/时延：最优平衡

#### 3. 摘要格式重组 ⭐
**工作量**：创建新模板，替换现有的"开头截断"模板

```md
## 📊 AI 定时分析摘要

**标的**: 600988 赤峰黄金
**评级**: 🟡观望
**评分**: 58/100

**综合结论**
短期超卖有反弹动能，但中期仍处空头结构，未突破 34.69 前不宜追高。

**操作建议**
- 短期：仅适合轻仓博弈反弹
- 中期：等待放量突破关键压力位再加仓
- 长期：关注金价周期与成本控制改善

**关键位**
支撑：29.02 | 压力：34.69 | 止损：28.50

**主要风险**
金价回落、成本抬升、财务披露不完整

> 完整报告请登录系统查看
```

**改进难度**：⭐ （20 行代码）

## 三、技术难点总结对比

| 难点维度 | 方案 A（增加字符数） | 方案 B（结论提取） |
|---------|-----------------|-----------------|
| **代码改动** | 1 行 | 50–200 行 |
| **数据库变更** | 无 | 无 |
| **外部依赖** | 无 | 可选 LLM 二次调用 |
| **结构化字段升级** | 无 | 需要，但关键 |
| **移动端适配** | ❌ 恶化 | ✅ 优化 |
| **信息聚焦度** | ❌ 一般 | ✅ 优秀 |
| **回归测试** | 简单 | 中等（需要覆盖多种报告格式） |
| **用户体验改善** | 无 | 显著 |

## 四、推荐方案：方案 B3（混合 fallback）

### 实施步骤

1. **升级结构化提取器**（quantia/lib/ai/report_parser.py）
   - 扩展 `_extract_rating()` 的正则库（+15 个同义词）
   - 升级 `_extract_advice()` 的文本清理逻辑
   - **预期效果**：成功率从 30% → 60%

2. **增加 LLM fallback**（quantia/job/stock_report_scheduled.py）
   ```python
   structured = extract_structured_fields(content)  # 一级提取
   if not structured.get('rating'):
       structured = extract_via_llm(content)        # 二级提取（LLM）
   ```
   - 成本：~0.01 CNY × 3 报告/天 = 0.03 CNY/天
   - **预期效果**：成功率从 60% → 98%

3. **新增摘要模板**（stock_report_scheduled.py）
   ```python
   def build_summary_payload(structured, original_report):
       return {
           rating: structured['rating'],
           score: structured['rating_score'],
           conclusion: structured['short_term_advice'][:100],
           advice: extract_key_points(original_report),
       }
   ```
   - **预期效果**：钉钉消息从"[截断的表格]"变成"[结论优先]"

4. **测试覆盖**
   - 测试 10+ 种报告格式（不同 prompt、不同 LLM）
   - 验证 fallback 链的覆盖率（应 > 95%）
   - A/B 测试用户体验改善

## 总结

**问题根因**：500 字符截断是为了移动端体验，但代价是丧失了报告的结论。

**最优方案**：用**结构化字段 + LLM fallback** 替换"报告前 N 字"的简陋方案，既保留移动端友好性，又确保关键信息（评级、建议、风险）一定在摘要里出现。

**建议采纳方案 B3**：实施成本低，改善幅度大，长期收益显著。
