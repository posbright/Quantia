### 0.2 方法论 → 现有能力映射（"选基"侧已覆盖）

| 方法论维度 | 落地字段 / 能力 | 现状 |

## 1. 问题拆解：入场信号 = 选基门槛 × 择时信号

$$
\text{可入场}(f, t) = \underbrace{\text{Quality}(f)\ge \theta_q}_{\text{选基：F7 评分/同类前列}}\ \wedge\ \underbrace{\text{Timing}(f, t)\ge \theta_t}_{\text{择时：本方案主体}}
$$

- **Quality(f)**：基金 f 是否"好"——直接读 `cn_fund_rank_score.score`（桶内百分位）+ 硬门槛（成立≥N 年、规模区间、回撤上限）。
- **Timing(f, t)**：t 时点是否"位置合适"——本方案要新建的核心。

场外基金**无 K 线/盘中价**，只有 T+1 每日净值，因此择时**只能基于净值序列 + 其跟踪标的（指数/ETF）**，这决定了候选方案的技术边界。

## 2. 择时方案候选（横向对比）

以下 5 套择时思路，均可用**现有表**实现，差异在数据依赖、稳健性、实现成本、可回测性。

### 方案 T1 —— 净值回撤入场（Drawdown-from-High）
- **信号**：基于 `cn_fund_nav_history.acc_nav`，计算当前相对滚动峰值的回撤 `dd = acc/peak - 1`。当 `dd ≤ -X%`（如 -8%/-15% 分档）视为"回调到位"，越跌越买。
- **优点**：纯净值即可，**零外部依赖**，对任意有净值历史的基金通用；直接复用 `scoring.compute_max_drawdown` 的口径。
- **缺点**：无估值锚，单边下跌中会持续"抄底"接飞刀；对趋势型基金误伤。
- **数据**：`cn_fund_nav_history`（已有）。

### 方案 T2 —— 净值动量/趋势过滤（NAV MA + 斜率）
- **信号**：`acc_nav` 对自身均线（如 60/120 日）上穿 + 斜率转正才允许入场；跌破长均线则不入场。
- **优点**：纯净值、避免接飞刀，"右侧"确认；与股票端 K 线均线思路一致，用户熟悉。
- **缺点**：右侧信号天然滞后，牛市初期踏空一段；均线参数敏感。
- **数据**：`cn_fund_nav_history`（已有）。

### 方案 T3 —— 跟踪标的估值分位择时（Valuation Percentile）★
- **信号**：把基金映射到其**基准/主投指数**（`cn_fund_profile.benchmark` 文本解析出主指数，或 `cn_fund_holding` 主行业→代表指数），取该**指数**的 PE/PB 历史百分位：`<30%` 低估可建仓、`>70%` 高估停买。指数数据**必须走 `load_benchmark_data`（AGENTS 规则 3），严禁 `load_stock_data`**。
- **优点**：**唯一有客观估值锚**的方案，最贴合"低买高卖"投资常识；对宽基/指数基金最有效。
- **缺点**：主动混合基金的基准解析不精确（基准是文本表达式）；指数 PE/PB 历史序列已确认**当前未落库**（见 §5 风险），需先新建 Fetch 管道与 `cn_index_valuation`。
- **数据**：指数行情（`load_benchmark_data`）+ 指数估值序列（**确认缺失，需新抓取**）+ `cn_fund_profile.benchmark`。

### 方案 T4 —— 定投/分批建仓（Cost-Averaging，弱化择时）
- **信号**：不预测点位，固定周期分批 or "越跌越加"的估值加权定投；直接产品化 [fund_backtest.py](../../quantia/core/fund/fund_backtest.py) 已验证的分批逻辑。
- **优点**：对普通用户最稳、最可执行；已有回测内核（`simulate_staged_buy` + `xirr`），落地成本最低。
- **缺点**：不是"点位"信号，是"节奏"策略；上涨市中总收益弱于一次性（平凡解，已在脚本免责中说明）。
- **数据**：`cn_fund_nav_history`（已有）。

### 方案 T5 —— 综合择时信号（T1 回撤 + T2 趋势 + T3 估值 加权）★★
- **信号**：把 T1/T2/T3 归一化到 0–100 后加权成 `TimingScore`，分档给出"低吸/定投/观望/高估勿追"标签；缺失维度（如无估值锚）自动降权重归一化——**与 `scoring.py` 桶内百分位 + 缺失填中性的既有范式完全一致**。
- **优点**：兼顾锚（估值）+ 位置（回撤）+ 确认（趋势），单维缺失可降级；可解释、可回测、可分档。
- **缺点**：实现最复杂；需要防止三维共线导致的"伪确认"。
- **数据**：T1/T2 纯净值（已有）+ T3 估值（确认缺失，首期自动降为 T1+T2 标签）。

### 2.1 横向对比表

| 维度 | T1 回撤 | T2 趋势 | T3 估值分位 | T4 定投 | **T5 综合** |
|------|:---:|:---:|:---:|:---:|:---:|
| 估值锚（低买高卖） | ✗ | ✗ | ★★★ | ✗ | ★★（含 T3 时） |
| 避免接飞刀 | ✗ | ★★ | ★ | ★★ | ★★ |
| 外部数据依赖 | 无 | 无 | 指数估值（确认缺失，需新抓取） | 无 | 可选 |
| 对普通用户可执行性 | ★ | ★ | ★★ | ★★★ | ★★ |
| 可回测性（现有内核） | ★★ | ★★ | ★ | ★★★ | ★★ |
| 实现成本 | 低 | 低 | 中高 | 低 | 中 |
| 与现有 scoring 范式一致 | ★★ | ★★ | ★ | ★★★ | ★★★ |
| 单维缺失可降级 | — | — | — | — | ★★★ |
| **稳健性/覆盖面综合** | 中 | 中 | 高但受限 | 中 | **最高** |

## 3. 选型结论：**T5 综合择时信号（以 T1+T2 为必备底座，T3 估值为可选增强）**，定投（T4）作为面向用户的执行封装

### 3.1 为什么是 T5（而非单一方案）
1. **稳健性最高、且可优雅降级**：T3 估值锚对宽基最强但对主动基金/数据缺失不可靠；T5 允许 T3 缺失时自动退化为 T1+T2 纯净值信号，覆盖**全部**有净值历史的基金——避免"只有一半基金能给信号"。
2. **与既有工程范式零冲突**：T5 的"多维归一化 + 桶内/序列百分位 + 缺失填中性 + 权重重归一化"与 [scoring.py](../../quantia/core/fund/scoring.py) 的做法**同构**，可直接复用其纯函数风格与测试范式，评审心智负担低。
3. **可回测**：T1/T2/T4 都能用 [fund_backtest.py](../../quantia/core/fund/fund_backtest.py) 的 `acc_nav` + `xirr` 内核回测"择时 vs 定投 vs 一次性"，用**真实 NAV** 出 IRR/回撤对比（严禁固定系数造伪 OOS，AGENTS 反模式）。
4. **T4 是执行层不是竞争项**：定投是"怎么买"，T5 是"该不该买/买多少"。最终产品 = **T5 给档位 → T4 给节奏**（低估档加大定投、高估档暂停），二者互补。

### 3.1bis 上线口径修订：三阶段策略，避免弱择时信号过度主导

鉴于 §6bis 已确认首期 T1/T2 属于弱依据净值技术信号、T3 指数估值数据又确认缺失，实际落地不能把"T5 最优"简单等同于"首期就用 TimingScore 主排序"。为避免把未经样本外验证的点位信号包装成强推荐，采用三阶段上线口径：

| 阶段 | 榜单/推送主口径 | TimingScore 角色 | 上线条件 |
|------|----------------|------------------|----------|
| **V1 可上线版** | **质量评分主排序**：每类基金先按 `quality_score` 取前列，再展示回撤/趋势/风险标签 | 仅作标签与仓位节奏提示，不参与或极低权重参与排序 | 现有 F7 + P1 基础 timing 即可 |
| **V2 验证通过版** | `final = 0.75*quality + 0.25*timing` 或"质量门槛 + timing 辅助排序" | 参与辅助排序，但不能覆盖质量门槛 | §6bis.4 walk-forward 样本外 go/no-go 通过 |
| **V3 理论最优版** | 质量 + 估值分位 + 定投节奏，宽基/指数基金提高 T3 权重 | T3 成为择时主锚，T1/T2 退为辅助确认 | 新建指数估值 Fetch 管道并完成历史验证 |

> **首期默认 V1**：每日精选榜语义是"同类中质量较优，且给出当前入场/定投节奏提示"，不是"系统建议立即买入"。只有 V2/V3 通过历史验证后，才允许文案升级为"择时辅助排序"。

### 3.2 最终入场信号定义（选基 × 择时）

```
EntrySignal(f, t):
    # 选基门槛（复用 F7，零新计算）
    q = cn_fund_rank_score.score(f)            # 桶内百分位 0-100
    pass_quality = q >= θ_q(默认70)
                   AND years_since(setup_date) >= 2
                   AND max_drawdown 优于桶内中位
                   AND scale in [下限, 上限]

    # 择时（新建，纯净值底座 + 可选估值）
    dd_score   = f(drawdown_from_high)         # T1，越跌分越高（有上限，防接飞刀）
    trend_score= f(nav_ma_slope)               # T2，站上长均线且斜率>0 加分
    val_score  = f(index_pe_pb_percentile)     # T3，可选；缺失→该维不参与，权重重归一化
    TimingScore = weighted_norm([dd_score, trend_score, val_score])  # 0-100

    tier = 低吸(≥75) / 定投(50–75) / 观望(30–50) / 高估勿追(<30)
    return pass_quality AND TimingScore, tier
```

> 上述 `EntrySignal` 是 V2/V3 的目标模型；首期 V1 不用 `TimingScore` 做硬过滤，只展示 timing 标签与定投节奏提示。

## 4. 落地实现方案（严守三管道分离与 AGENTS 规则）

> 全部为 **Analysis/Web 只读 MySQL + 纯计算**，**不新增任何外部 API 调用**（规则 1）。
> 指数数据一律 `load_benchmark_data`（规则 3）。SQL 显式列 + 校验存在（规则 7）。
> 新前端调用必须在 `web_service.py` 注册路由（规则 8）。

### 4.1 核心纯函数模块（新增）
`quantia/core/fund/timing.py`（与 `scoring.py`、`fund_backtest.py` 平级，**纯函数、无 I/O、可合成数据测试**）：
- `drawdown_from_high(acc_nav, lookback) -> float`（复用 `scoring._clean_nav` 口径）
- `nav_trend_score(acc_nav, ma_window) -> float`
- `valuation_percentile_score(index_pe_or_pb_series) -> float`（输入已由上层取好的指数估值序列）
- `compose_timing_score(dd, trend, val, weights) -> {score, tier, components}`（缺失维度重归一化，模仿 scoring 缺失填中性）

### 4.2 择时批量 job（可选，新增）
`quantia/job/analysis_fund_timing_job.py`：读 `cn_fund_nav_history`（+ 可选指数估值）→ 逐基金算 `TimingScore` → 写新表 `cn_fund_timing_score`（主键 `(date, code)`，`chunksize=500`，NaN/inf 源头清洗）。与 `analysis_fund_score_job.py` 同构。
> 若不想加每日 job，可退化为 Web 端按需实时计算（单基金净值序列量小，实时可接受）——见 §6 分期。

### 4.3 Web Handler（新增）
`quantia/web/fundTimingHandler.py`：`GET /quantia/api/fund/timing?code=` → 返回 `{timing_score, tier, components:{dd,trend,val}, quality_pass, as_of, disclaimer}`。只读表，**禁"买/卖/加仓"措辞**（沿用 F13 免责规范，`labels.py`）。在 `web_service.py` 注册路由（规则 8）。

### 4.4 前端（增量）
`views/fund/` 详情抽屉新增"入场时机"卡片：档位徽章（低吸/定投/观望/高估）+ 三维分量条 + 回撤/估值分位小图 + 免责声明；**移动端做卡片式适配**（AGENTS 移动端规范）。列表页可加"当前处低吸档"过滤。

### 4.5 回测与验证（复用现有内核）
用 [fund_backtest.py](../../quantia/core/fund/fund_backtest.py) 扩展对照实验：**T5 择时买入 vs 纯定投 vs 一次性**，同窗口同基金池，输出中位总收益 / **IRR** / **最大回撤**，用真实 `acc_nav` 切样本内外，验证"择时是否真降回撤/提 IRR"。落 `document/fund/` 实验结论。

## 5. 风险与前置核实（验证优先，动手前必查）

1. **指数估值序列已确认缺失**（T3 前提，2026-07-08 核实）：全库仅有**个股级**估值（`cn_stock_spot.pbnewmrq`、`cn_stock_cpbd.PE_TTM/PB_MRQ_REALTIME`），**没有指数级 PE/PB 历史分位表**。故 **T3 首期不可得 → 首期只上 T1+T2（T5 自动降级到二维）**。T3 作为后续增量，其估值序列抓取**必须走 Fetch 管道**（新爬虫 + 新表，如 `cn_index_valuation`），不能塞进 Handler（规则 1）。
2. **基准文本解析可靠性**：`cn_fund_profile.benchmark` 是"沪深300×80%+..."文本，解析主指数有噪声 → 首期主动基金可退化用 `cn_fund_holding` 主行业代表指数，或直接跳过 T3。
3. **幸存者/覆盖偏差**：`cn_fund_nav_history` 仅回填 Top-N，回测结论会偏乐观（脚本已有免责）→ 结论页必须同步披露。
4. **接飞刀风险**：T1 单用会在单边下跌持续抄底 → T5 必须让 T2 趋势维度对"跌破长均线"扣分，形成保护。
5. **合规**：全程"信号/档位"措辞，非投资建议，带风险提示（对齐 F13）。

## 6bis. 方案复审：是否最优 / 是否有依据 / 能否经受历史验证

> 应用户要求对本方案做批判性复审。以下结论**据实标注可信度**，不粉饰。

### 6bis.1 是否有依据（学术/经验证据强弱）

| 信号 | 依据强弱 | 说明 |
|------|:---:|------|
| **选基：多因子截面排序（F7）** | **强** | 动量/低费率/规模/风险调整收益是公募选基的成熟因子，桶内截面排序稳健，已落地。 |
| **T3 估值分位择时（指数 PE/PB）** | **强** | "低估值分位买入、高估值分位停手"对宽基有大量历史证据（均值回归）。**但本库暂无该数据**（§5.1），首期用不上——这是本方案最大的"证据强但数据缺"矛盾。 |
| **T4 定投/分批** | **中强** | 对普通投资者降低择时风险的证据充分；但"提高总收益"无证据（上涨市必输一次性，平凡解）。价值在**降回撤/纪律**，不在超额。 |
| **T1 净值回撤入场** | **弱** | "基金净值自身回撤 X% 后反弹"缺乏稳健证据；对主动基金，回撤可能是**经理能力衰退**而非均值回归 → 越买越亏的尾部风险真实存在。 |
| **T2 净值均线趋势** | **弱—中** | 时序动量在指数上有一定证据，但直接套在**单只基金净值**上样本相关性高、易过拟合。 |

**诚实结论**：本方案**证据最强的两块（选基 F7、估值择时 T3）**——前者已建成、后者数据缺失。首期能立即交付的 T1+T2 属于**弱依据的净值技术信号**。因此首期择时应**明确定位为"风险提示/仓位参考"，而非"精确择时"**，主执行手段落在证据更稳的**定投（T4）**上。这一定位需写进前端文案与推送免责，避免夸大。

### 6bis.2 是否为最优解

- **框架层面最优**："复用已建成选基 + 新建择时 + 缺维降级"避免重复造轮子、覆盖面最大，确为当前约束下的最优骨架。
- **信号层面有保留**：受限于"指数估值数据缺失"，首期最优只能退到 T1+T2，这**不是理论最优（T3 才是）**，是**数据约束下的次优**。故把 T3 提前到**与 P1 并行的 Fetch 预研**，比排到 P3 更合理——见下方分期修订建议。
- **更优的替代定位**：若追求"经得起历史检验"，**首期主推 T4 定投信号 + T1/T2 仅作辅助提示**，比"主推 T1/T2 点位择时"更稳、更诚实。

### 6bis.3 能否经受住历史验证（回测方法学红线）

T1/T2 的阈值（回撤 -8%/-15%、均线 60/120）都是**自由参数**，直接在全样本调参必然过拟合、回测虚高。要"经受住历史验证"必须遵守：

1. **样本外 / Walk-forward**：参数在训练段选定，检验段只读评估；严禁全样本调参后报回测（对齐 AGENTS "禁止固定系数造伪 OOS"）。
2. **三方对照**：T5 择时 vs 纯定投(T4) vs 一次性(lump-sum)，同池同窗，比 **IRR + 最大回撤**（不比总收益——总收益是平凡解）。
3. **幸存者偏差披露**：`cn_fund_nav_history` 仅回填 Top-N，结论系统性偏乐观，必须在结论页显式声明。
4. **判定门槛**：若 T1/T2 择时在样本外**不能同时改善 IRR 或显著降低回撤**，则**首期不上点位择时**，只保留定投 + 回撤风险提示。这是"能否经受历史验证"的硬性 go/no-go。

### 6bis.4 可执行历史验证协议（go/no-go）

为避免全样本调参和事后归因，择时信号上线为排序权重前必须按固定协议验证：

| 项 | 规则 |
|----|------|
| 样本池 | 每个 `fund_type` 单独验证；剔除货币型点位择时；仅使用当时已披露且 `acc_nav` 覆盖满足最小点数的基金 |
| 日期口径 | 所有输入必须满足 `data_as_of <= decision_date`；回测时禁止使用计算日之后补录数据 |
| 训练/测试 | 滚动 walk-forward：训练 3 年、测试 1 年，逐年或逐半年推进；参数只在训练段选择 |
| 参数网格 | 回撤阈值、均线窗口、趋势斜率阈值必须预注册；不得根据测试集二次调参 |
| 对照组 | ① 一次性买入；② 月定投；③ 质量 Top10 不择时；④ T1/T2/T5 择时版本 |
| 指标 | 以 **IRR、最大回撤、Calmar、中位数/25分位表现** 为主；总收益仅作参考（上涨市总收益偏向一次性，是平凡解） |
| 通过门槛 | 样本外相对"质量 Top10 不择时"：最大回撤中位数下降 ≥10%，且 IRR 中位数不低于基准；若只降回撤但 IRR 明显受损，只能作为"风险提示"不上排序 |
| 失败降级 | 未通过则 V1 保持：质量主排序 + timing 标签；不允许在钉钉文案中出现"入场机会"强措辞 |

> **复审最终建议**：保留总体框架；把首期定位从"点位择时"下调为"**定投主执行 + 弱择时提示**"；T3 估值 Fetch 预研与 P1 并行；点位择时以本节样本外 go/no-go 决定是否从 V1 升级到 V2/V3。

## 7. 每类基金 Top10 综合精选榜 + 每日钉钉推送

> 用户新增需求：**综合选基+择时，按评分选出每类基金前 10 名展示，每日钉钉推送，点链接看详情**。
> 全部为 **Analysis 只读 MySQL + 计算 → 落库 → Notification 出口**，无外部行情 API（规则 1）。

### 7.1 综合评分与"每类 Top10"口径

严守既有硬约束：**强制按 `fund_type` 分桶，不跨类型混排**（货币 7 日年化 vs 股票型年收益不可比）。每个 `fund_type` 桶内取前 10。

榜单口径三种，**对比择优**：

| 口径 | 定义 | 优点 | 缺点 |
|------|------|------|------|
| **A. 加权和** | `final = w_q·quality + w_t·timing`（如 0.6/0.4） | 单一分值、易排序 | 差择时可被高 quality 淹没，可能推"好但正高估"的基金，误导入场 |
| **B. 门槛+排序（V2/V3 推荐）** | 先 `quality≥θ_q ∧ 硬约束(成立≥2y/规模区间/回撤优于中位)` 过滤，再按通过验证的 timing 分桶内排序取 Top10；timing 缺失的基金排在有信号之后 | 与本方案"入场信号=选基门槛×择时"定义一致；不会把"高估的好基金"当"该买"推给用户 | 需要 timing 有效且通过样本外验证，首期不应直接启用 |
| **C. 质量主排序 + 择时标签（V1 推荐）** | 每类基金先按 `quality_score` 排序取 Top10；TimingScore 只作为"回撤/趋势/定投节奏"标签展示，不决定主排序 | 证据最稳、误导风险最低，可立即上线；与 §6bis 的弱择时定位一致 | 不是"到位置"榜，而是"好基金 + 当前位置提示"榜 |

**首期选定 C（质量主排序 + 择时标签）**：它与 §6bis 的证据强弱结论一致，避免用未验证的 T1/T2 主导每日推送。待 §6bis.4 样本外验证通过后，再升级到 B 或 `final = 0.75*quality + 0.25*timing` 的 V2 口径；待 T3 指数估值接入并验证后，升级到 V3。

### 7.1bis 分基金类型策略口径

不同 `fund_type` 的净值波动、披露滞后和择时有效性差异很大，不能用同一套 T1/T2 权重机械套用：

| 基金类型 | 首期 V1 排序 | Timing 标签 | 备注 |
|----------|--------------|-------------|------|
| 股票型 / 混合型 | `quality_score` 主排序 | 可展示 T1 回撤、T2 趋势、定投节奏 | 后续最适合接 T3 估值锚 |
| 指数型 | `quality_score` + 跟踪误差/费率（如后续补列） | 首期 T1/T2 标签；V3 优先接指数估值 T3 | 最适合估值分位择时 |
| 债券型 | 夏普/回撤/波动/费率优先 | T1/T2 仅作风险标签，权重显著降低 | 波动小，净值均线容易变成噪音 |
| 货币型 | 七日年化、万份收益、费率、规模/稳定性 | **不做点位择时** | 不应推"低吸/高估"措辞 |
| QDII | 质量主排序 + 风险标签 | 注意海外市场交易日与汇率，`data_lag_days` 必须展示 | 净值披露更滞后，严控 as-of |
| FOF | 质量主排序 + 回撤/波动标签 | 慎用单基金净值趋势择时 | 更像配置产品，趋势信号解释性弱 |

### 7.2 落库（新表，供前端 + 推送共用）

`TABLE_CN_FUND_DAILY_PICK`（主键 `(date, fund_type, code)`，`chunksize=500`，NaN/inf 源头清洗）：

| 列 | 说明 |
|----|------|
| date | 计算日（运行日） |
| fund_type | 基金类型桶 |
| rank_in_type | 桶内名次 1..10 |
| code / name | 基金代码 / 简称 |
| quality_score | 复用 `cn_fund_rank_score.score` |
| timing_score / timing_tier | T5 择时分 + 档位（低吸/定投/观望/高估） |
| final_score | 排序用综合分；V1 = `quality_score`，V2 = `0.75*quality + 0.25*timing` 或通过验证后的辅助排序键 |
| max_drawdown / rate_1y | 榜单展示辅助列 |
| score_as_of / nav_as_of | 质量评分与净值数据各自的可见日期 |
| data_lag_days | `date - nav_as_of`，用于提示净值披露滞后 |

> 由 `analysis_fund_pick_job.py`（Analysis 管道）生成：读 `cn_fund_rank_score` + `cn_fund_nav_history` 算 timing → 首期按 §7.1 口径 C 每桶取 Top10 → 落库。**评分为排序辅助，禁落为可交易 strategy_template**（对齐 F7/Phase 9 约束）。

### 7.2bis as-of 日期对齐与未来函数防线

每日精选和历史回测必须用"当时可见信息"，否则会引入未来函数：

- `score_as_of = MAX(cn_fund_rank_score.date) <= pick_date`。
- `nav_as_of = MAX(cn_fund_nav_history.nav_date) <= pick_date`，且落 `data_lag_days = pick_date - nav_as_of`；QDII/节假日滞后需在前端提示。
- 若基金 `data_lag_days` 超过阈值（如 5 个自然日），可降权或标注"净值滞后"，避免把陈旧数据当今日信号。
- 历史回测时只允许使用 `decision_date` 之前已披露的数据；若后续补录历史净值，回测要按 `nav_date` 与披露滞后规则重放，不能直接用最终全量表。

### 7.3 每日钉钉推送（对齐既有推送链路，不新造轮子）

完全复用现有基础设施（已核实）：
- `DingTalkChannel.build_markdown_payload(title, markdown)` + `_load_config(user_id, notify_type, 'dingtalk')` + 签名 URL；
- 用户偏好门控 `push_enabled`（同 `push_report_summary_to_dingtalk` 先例）；
- 详情链接用 `QUANTIA_WEB_BASE_URL`（未配置回退 `http://<host>:9988`）拼 `_build_*_url`。

**通知配置与幂等**：
- 新增 notify type：`fund_daily_pick`；默认关闭，由用户在通知设置中启用，避免未经同意每日触达。
- 按用户配置逐个推送；不同用户可绑定不同钉钉 webhook。
- 新增发送日志或复用现有通知表，幂等键建议为 `(user_id, notify_type, pick_date)`，同一用户同一日最多成功推送一次。
- 发送失败记录错误并可人工重试；自动重试需有次数上限，禁止在钉钉失败时循环刷屏。

**推送内容设计（防止刷屏 + 钉钉限流 20 条/分、单条 markdown 体积限制）**：
- **紧凑摘要**，不逐条塞 60 只。每个 `fund_type` 桶只展示 **Top3**（代码/简称/quality/timing 档位徽章），每只带独立详情链接；
- 末尾一条"[查看完整每类 Top10 榜单]"链接跳前端榜单页；
- 标题如 `📈 每日基金精选榜 2026-07-08`；
- **非买卖措辞**，带风险免责（对齐 F13）。

**详情链接（重要增量，已核实缺口）**：现基金页 `views/fund/index.vue` 用抽屉 `FundDetailDrawer` + `openDetail(row)`，**不读 `route.query.code`**。因此推送里的每只基金链接 = `{{BASE}}/#/fund/rank?code=XXXXXX`，需给基金页加**深链支持**：`onMounted/onActivated` 读 `route.query.code` → 自动 `openDetail`。整榜链接 = `{{BASE}}/#/fund/rank?pick=1`（打开"每日精选"分区）。此改动是"点链接看详情"可用的前提，必须与推送同期交付（规则 8：前端调用/深链要有对应支撑）。

### 7.4 调度

在 `cron/cron.workdayly/` 增脚本，**顺序 = 基金抓取 → `analysis_fund_score_job`（F7）→ `analysis_fund_pick_job`（榜单+timing）→ 钉钉推送 job**。推送 job 只读 `cn_fund_daily_pick` + 发 webhook，失败重试/降级不影响其他 job。

### 7.5 前端（增量）

`views/fund/` 加"每日精选"分区/Tab：按 `fund_type` 分组展示各桶 Top10（quality + timing 档位徽章 + 回撤/近1年），点行开详情抽屉；`route.query.code` 深链自动开抽屉、`route.query.pick` 定位精选分区。**移动端卡片式适配**（宽表≥5列，AGENTS 移动端规范）。

### 7.6 该功能的分期落点

| 期 | 内容 |
|----|------|
| **P5** | `analysis_fund_pick_job` + `cn_fund_daily_pick` 表（V1 口径 C：质量主排序 + 择时标签）；前端"每日精选"分区 + `?code=` 深链自动开抽屉 |
| **P6** | 每日钉钉推送 job（复用 `DingTalkChannel` + `push_enabled` 门控 + 紧凑摘要 + 详情链接）+ cron.workdayly 编排 |
| **P7** | 通过 §6bis.4 样本外验证后，才允许将 timing 升级为辅助排序权重；若不通过则维持 V1 |

> 依赖 P1（timing.py）先落地；首期榜单明确为"好基金+当前位置提示"，待 §6bis.4 go/no-go 通过后再强化 timing 权重。

## 8. 一句话总结

> **选基**复用已建成的 F7 多因子评分（好基金门槛 `score≥70` + 成立/规模/回撤硬约束，证据强）；
> **择时**新建 `TimingScore = 回撤(T1) + 趋势(T2) + 估值分位(T3, 数据缺失暂缓)` 的**综合信号 T5**，缺维自动降级；
> **诚实定位**：首期只有弱依据的 T1+T2，故以**定投(T4)为主执行、择时为风险提示**，点位择时须过"样本外 go/no-go"才上线；
> **每日精选**按 `fund_type` 分桶首期用"**质量主排序 + 择时标签**"口径取每类 Top10，落 `cn_fund_daily_pick`，
> 复用既有 `DingTalkChannel` + `push_enabled` 每日紧凑推送，链接经**基金页 `?code=` 深链**打开详情抽屉（需同期加深链支持）。
