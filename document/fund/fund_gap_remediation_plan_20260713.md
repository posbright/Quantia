# 基金每日精选差距修复方案（审核稿）

> 日期：2026-07-13
> 状态：**P0-A / P0-B 已完成；P1-B 历史删除缺陷已止损；P1-A、P1-B 补链、P2、P3 待分阶段实施**
> 对照文档：[good_fund_selection_and_entry_timing_plan.md](good_fund_selection_and_entry_timing_plan.md)

## 1. 目标与边界

本方案只处理现网核实存在的差距，并保持三条硬边界：

1. Fetch 管道负责外部数据；Analysis/Web 只读 MySQL 与本地缓存。
2. V1 继续采用“质量主排序 + timing 弱标签”，样本外验证通过前不让 timing 参与排序。
3. 缺数据必须显式降级，不把“未知”误判为“可申购”“已更新”或“风格稳定”。

本文最初作为审核稿编写；当前 P0-A、P0-B 与 P1-B 数据删除止损已完成。其余工作仍按独立阶段实施，
每阶段单独审核、测试、部署、黑盒验证和提交，禁止一次性大改。

## 2. 生产核验证据

核验均基于只读 SQL；申购状态仅调用一次 `akshare.fund_purchase_em()` 后在内存中与精选榜关联，未落库。

### 2.1 已证实的真实问题

| 项目 | 生产证据 | 判定 |
|---|---|---|
| 申购/赎回/限额状态 | `cn_fund_purchase_status` 不存在；2026-07-10 最新 70 条精选中，52 条开放申购、18 条限大额；QDII Top10 全部限大额，其中多只日限额仅 100 元 | **真实硬缺口** |
| 异常截面门控 | 评分、精选、推送在同一次 `run_analysis` 中顺序执行；评分失败后脚本仍继续精选和推送；精选会自动读取 `<= pick_date` 的旧评分 | **真实链路风险** |
| 净值历史冷启动 | 最新各类型 Top50 中缺历史：混合型 14、指数型 1、债券型 31；最新精选中 6 只债券型无历史（货币型另有 10 只，按设计不做 timing） | **已发生的数据缺口** |
| 样本外验证 | 已有 timing 纯函数和基础定投回测内核，但没有 walk-forward 作业、结果表或通过结论 | **V2/V3 升级阻塞项** |

### 2.2 当前没有发生、但缺少防线的问题

最新 `cn_fund_rank` 快照为 2026-07-10，共 20,486 行，是前 5 个快照行数中位数的 100.21%。其中 95.06% 的 `nav_date` 不早于 2026-07-09，当前截面并未失真。因此不能把“本次数据不完整”作为现状结论；真实问题是**没有自动阻止未来异常截面继续生成和推送榜单**。

### 2.3 审查中已排除的误判

- V1 展示 timing 标签符合原计划，不等于违反 go/no-go；违规边界是“未验证就参与排序或使用强推荐措辞”。
- 科创板 `688` 已进入 A 股识别与行业补全路径，不再是当前主阻塞。
- 风格漂移不是算法/前端未实现，而是旧季度被抓取保存逻辑删除。
- `?pick=1` 和 `?code=` 深链已经在基金中心处理，不需要重复开发。

## 4. P0-A：申购状态全链路

### 4.1 方案比较

**方案 A：每天保存全市场历史快照**

- 优点：可做完整历史回溯。
- 缺点：约 2.7 万行/日，年增约 650 万行；对当前 1.6 GB 服务器成本过高。

**方案 B：只保存当前状态**

- 优点：表小、查询快。
- 缺点：无法追溯某日精选当时的申购状态。

**选定方案 C：当前状态表 + 精选日快照**

- `cn_fund_purchase_status` 只保留每只基金最新状态。
- `cn_fund_daily_pick` 快照入选当日状态，约 70 行/日，可审计且容量可控。
- 不保存全市场每日历史；后续若回测需要，再建设候选池级历史，不提前扩容。

### 4.2 数据模型

新增 `TABLE_CN_FUND_PURCHASE_STATUS`，主键 `code`：

| 字段 | 类型 | 含义 |
|---|---|---|
| `code` | VARCHAR(6) | 基金代码 |
| `name` | VARCHAR(60) | 数据源基金简称 |
| `purchase_status` | VARCHAR(20) | 原始申购状态 |
| `redemption_status` | VARCHAR(20) | 原始赎回状态 |
| `next_open_date` | DATE NULL | 下一开放日 |
| `min_purchase` | FLOAT NULL | 购买起点 |
| `daily_limit` | FLOAT NULL | 日累计限定金额 |
| `fee` | FLOAT NULL | 手续费 |
| `fetched_at` | DATETIME | 成功抓取时间 |

`cn_fund_daily_pick` 增加快照字段：

- `purchase_status`
- `redemption_status`
- `daily_limit`
- `purchase_availability`：`available / limited / unavailable / unknown`
- `purchase_as_of`

所有新表必须在 `quantia/core/tablestructure.py` 注册；写入走统一 DB 层，`chunksize=500`，数值列在源头清理 NaN/inf。

### 4.3 Fetch 设计

1. 在 `quantia/core/crawling/fund_em.py` 增加列映射纯函数和 `fund_purchase_status_all()`。
2. 新增 `quantia/job/fetch_fund_purchase_status_job.py`，单次抓全市场后整批 upsert。
3. 仅成功获取非空、列契约完整的数据时更新；源失败时保留旧状态，禁止清空表。
4. 每次记录行数、状态分布、抓取时间；异常状态值保留原文并映射为 `unknown`。
5. 接入 `run_fetch`，位于基金排名之后、分析之前；失败为 warn，但会让后续精选显示旧/未知状态，不伪装为开放。

### 4.4 申购可用性规则

规则集中在纯函数模块，禁止在 job、Handler、Vue 各写一套：

- 明确“开放申购” → `available`。
- 明确“限大额” → `limited`，保留入选，展示限额；**不因 100 元或 1,000 万元使用任意阈值改变质量排序**。
- 明确暂停、封闭或停止申购 → `unavailable`，从精选候选中剔除。
- 缺表、缺行、状态无法识别、数据超过 2 个自然日 → `unknown`，保留但必须显示“申购状态未知”。

选择时先按现有质量规则取足候选，再剔除 `unavailable` 并向后补位。若某桶不足 10 只，允许少于 10，禁止把不可申购基金补回。`limited` 与 `unknown` 不改变 V1 的质量顺序。

此规则避免两类错误：把“限大额”误当完全不能买导致 QDII 榜单清空；把抓取失败误当开放申购。

### 4.5 前端和推送

- 每日精选：展示“开放/限额/暂停/未知”状态；限额显示格式化金额。
- 详情抽屉：展示申购、赎回、下一开放日、起购金额、日限额和数据时间。
- 钉钉：`limited` 添加简短限额标志；`unknown` 标注状态待核实；`unavailable` 正常情况下不应进入推送。
- 所有措辞为状态事实，不使用“建议买入”。

### 4.6 验收标准

- 源映射覆盖正常、缺列、未知状态、NaN/inf。
- Fetch 重跑幂等；源空结果不清表。
- 精选中的暂停申购基金为 0；限大额基金保留且金额正确。
- API 与桌面/移动端均能区分四种状态。
- 用当前 70 条精选做黑盒复核，预期能识别现有 18 条“限大额”。

## 5. P0-B：截面完整度与整链门控

### 5.1 不采用的方案

- **只把 cron 再推迟**：无法防接口局部失败、空数据或旧数据回退。
- **只检查总行数**：行数完整不代表 `nav_date` 已更新。
- **只检查 `nav_date == snapshot_date`**：凌晨任务的正常目标是上一交易日，QDII/FOF 还有天然延迟，会误报。

### 5.2 选定的双门槛

在 `analysis_fund_score_job` 计算前执行只读检查，返回结构化结果：

1. **规模门槛**：最新快照行数 / 前 5 个有效快照行数中位数 ≥ 90%。
2. **新鲜度门槛**：
   - 目标净值日取 `cn_stock_trade_date` 中 `< snapshot_date` 的最近交易日；
   - 股票型、混合型、指数型、债券型、货币型纳入硬门槛；
   - `nav_date >= target_nav_date` 的比例 ≥ 90%；
   - QDII、FOF 单独统计并告警，不进入首版硬门槛，避免天然 T+1/T+2 延迟误阻断全链。
3. **日期门槛**：评分使用的 rank 快照必须等于本次期望快照；禁止静默读取更老快照。
   - `run_workdayly` 显式注入同日模式，期望 `<= T` 的最近交易日（即 T 日）；
   - 独立 `run_analysis` 保持 T+1 模式，期望 `< 执行日` 的最近交易日；
   - 周末、节假日统一通过交易日历回退，禁止按自然日减一；同日抓取失败留下旧快照时日期门槛必须拒绝。

阈值由环境变量配置但默认 0.90；日志必须打印分子、分母、目标日、各类型分布和失败原因。

### 5.3 链路行为

- 看门狗不通过：评分不写库，返回专用“数据未就绪”退出码。
- `run_analysis` 必须显式判断评分结果；评分失败或未就绪时，**跳过当日 pick 和 push**，其他独立分析任务不受影响。
- 禁止精选自动退回旧 `score_as_of` 后仍以当天日期落榜/推送。
- 可选补偿任务在 06:30 再重试一次；重试仍失败则当天不发，运维可见告警。
- 手工 `--date` 回放必须显式提供 `--allow-stale`，默认仍受门控；生产 cron 不设置该参数。

实现补充：历史评分按指定日期读取精确 `cn_fund_rank` 截面，且净值、画像、持仓均限制为
`nav_date/update_date <= score_date`；未来抓取的申购状态降级为 `unknown`。06:30 补偿先按
`score_as_of` 查既有榜单，有榜只补该日期通知，无榜且门禁通过才重建，避免同日/次日重复榜。

### 5.4 验收标准

- 合成 89% 与 90% 边界测试。
- 周一凌晨正确取上周五为目标日；节假日使用交易日历而不是自然日减一。
- QDII/FOF 滞后不会单独阻断。
- 评分异常、数据未就绪、评分为空三种情况都不会执行 pick/push。
- 当前 2026-07-10 数据应通过：规模比 100.21%，核心类型新鲜度高于 90%。

## 6. P1-A：Top50 净值历史补链

### 6.1 方案比较

- 扩大固定 `QUANTIA_FUND_NAV_TOPN`：实现简单，但仍按近一年收益选，不保证质量 Top50；还会重复抓大量无关基金。
- 评分 job 直接调用 Fetch：违反 Fetch/Analysis 分离，否决。
- **选定：Analysis 只写请求队列，Fetch worker 消费**。

### 6.2 队列表

新增 `cn_fund_nav_backfill_queue`：

- `code` 主键
- `fund_type`
- `reason`（首版固定 `quality_top50`）
- `priority`
- `status`（pending/running/succeeded/retry/failed）
- `requested_at`、`started_at`、`finished_at`
- `attempts`、`next_retry_at`、`last_error`

Analysis 评分完成后只做幂等 upsert：非货币类型各桶 Top50 中，无任何净值历史或最新历史超过 7 天者入队。货币型明确排除；债券型不再因当前默认排除策略被永久遗漏。

Fetch worker 复用现有单基金净值抓取与增量写入函数，限制单批数量、总时长、重试次数和退避。状态从 running 恢复需有超时租约，避免进程中断后永久卡死。

### 6.3 验收标准

- 重复评分不产生重复队列行。
- worker 失败不删除已有净值，成功后标记 succeeded。
- 非货币新入 Top50 在 2 小时 SLA 内完成或进入可解释的 retry/failed。
- 当前已知缺口应被识别：混合型 14、指数型 1、债券型 31；不把货币型 50 计为缺陷。

## 7. P1-B：持仓季度历史保留与风格漂移点亮

### 7.1 根因修复

当前主键已是 `(code, quarter, stock_code)`，天然支持多季度；问题仅在保存前按 code 全删。修改为：

1. 抓到某一季度后，只删除/覆盖 `(code, quarter)`，不得删除该 code 的其他季度。
2. 抓取映射函数支持返回最近两期，而不是先裁成最新一期。
3. 首次补链按基金分批抓最近两期；后续每季度增量维护。
4. 空响应和瞬态失败都不得删除历史季度。
5. 保留最近 8 个季度即可，清理必须另设受控任务并遵守删除确认规则；本阶段不自动批量删除。

### 7.2 验收标准

- 同一基金可同时保留至少两个季度。
- 重抓当前季度不会影响上一季度。
- Handler 对两期已分类持仓返回 `drift_available=true`。
- 无上一季、上一季全未分类时继续显式不可用。
- 生产补链后报告“可计算漂移基金数”，不能只报告总持仓覆盖。

## 8. P2：Walk-forward go/no-go

### 8.1 定位

该阶段只决定是否允许从 V1 升级，不自动修改生产排序权重。结果必须经人工审核。

### 8.2 数据与防偏差

- 每个 `fund_type` 单独验证；货币型排除。
- 只使用 `decision_date` 当时可见的数据，禁止用后补 profile/holding/purchase 状态制造历史真相。
- 当前 `cn_fund_nav_history` 是按现时 Top-N 回填，存在幸存者偏差；首版报告必须单列覆盖率和偏差，不满足最低历史覆盖时结论为 inconclusive，不能判 pass。
- 参数网格预注册；训练 3 年、测试 1 年，逐半年滚动。

### 8.3 对照组与指标

对照组：一次性、月定投、质量 Top10 不择时、T1+T2、T1+T2+T3。统一资金、费用、决策日和基金池。

主指标：OOS IRR、最大回撤、Calmar、中位数、25 分位；总收益仅作辅助。

通过条件沿用原文档：相对质量 Top10 不择时，OOS 最大回撤中位数下降 ≥10%，且 IRR 中位数不低于基准。按类型分别给出 `pass / fail / inconclusive`，禁止用全类型平均掩盖失败桶。

### 8.4 产物与审批

- 可复现的研究脚本/作业和纯函数测试。
- 固化参数、代码版本、数据截止日、样本覆盖和结果文档。
- 不新增自动读取“最近一次 pass 就改权重”的生产开关。
- 只有人工批准某个 fund_type 后，另开 V2 实施任务；未批准类型保持 `final_score=quality_score`。

## 9. P3：timing 档位过滤（可选）

在 V1 下该控件必须明确是“当前位置标签筛选”，不是买入推荐：

- 使用分段控件：全部、低吸、定投、观望、高估勿追、暂无数据。
- 默认“全部”，不默认隐藏高估或无数据基金。
- 桌面和移动端共用同一 computed 数据源；切桶后保留或明确重置筛选。
- URL 可选同步 `timing_tier`，但不得破坏已有 `?pick=1&code=` 深链。
- 增加组件测试：单选、全部、NULL 档位、切桶、移动卡片数量。

## 10. 实施顺序与提交边界

每个阶段独立提交，前一阶段验收通过后再进入下一阶段：

1. P0-A 申购状态：crawler + table + fetch job → 精选快照 → API/前端/推送。
2. P0-B 完整度门控：纯函数 + score gate → cron 链路门控 → 告警/补偿。
3. P1-A 净值补链：队列表 + enqueue → Fetch worker + cron。
4. P1-B 持仓历史：抓取/保存语义 → 两期补链 → 风格漂移黑盒验证。
5. P2 回测：研究实现 → 结果报告 → 人工 go/no-go。
6. P3 档位过滤：仅在产品审核确认措辞后实施。

禁止把 P2 结果未审核就与生产权重改动放在同一提交。

## 10.1 2026-07-13 深审结论与优化升级

| 阶段 | 当前事实 | 风险/缺口 | 优化后的实施边界 |
|---|---|---|---|
